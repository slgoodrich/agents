# Specification Techniques

## Overview
Methods and frameworks for creating clear, actionable product specifications that align teams and reduce ambiguity.

## Specification Formats

### Product Requirements Document (PRD)
- **Problem** - What we're solving
- **Solution** - How we'll solve it
- **Requirements** - What we need to build
- **Success metrics** - How we measure success
- **Dependencies** - What else is needed

### User Stories
**Format**: As a [user], I want [capability] so that [benefit]

**Acceptance Criteria**:
- Given [context]
- When [action]
- Then [outcome]

### Job Stories
**Format**: When [situation], I want to [motivation], so I can [expected outcome]

### Use Cases
- **Actor** - Who interacts
- **Preconditions** - Starting state
- **Main flow** - Happy path
- **Alternative flows** - Variations
- **Postconditions** - End state

## Writing Clear Requirements

### INVEST Criteria (User Stories)
- **Independent** - Can be developed separately
- **Negotiable** - Details can be discussed
- **Valuable** - Delivers user/business value
- **Estimable** - Can be sized
- **Small** - Fits in sprint
- **Testable** - Clear acceptance criteria

### Specification Best Practices
1. **Start with why** - Context and rationale
2. **Define success** - Measurable outcomes
3. **Be specific** - Eliminate ambiguity
4. **Include examples** - Illustrate behavior
5. **Cover edge cases** - Handle exceptions
6. **Identify dependencies** - Technical and business
7. **Visual aids** - Diagrams, mockups, flows

## Edge Cases and Error Handling
- **Validation** - Invalid inputs
- **Permissions** - Unauthorized access
- **Limits** - Capacity constraints
- **States** - Empty, partial, complete
- **Failures** - System errors, timeouts
